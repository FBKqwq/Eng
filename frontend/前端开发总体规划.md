# 前端开发总体规划：页面体系、组件架构与智能体可视化耦合设计

> 项目名称：基于 ELK、Kafka 与 LangGraph 的模拟电商实时日志分析与智能异常诊断系统
> 文档定位：`location/frontend/` 下一阶段开发的总体规划文档，回答三个核心问题：
> 1. 按后端规划（`backend/doc/后端开发总体规划-Services-LangGraph-MCP.md`、`backend/doc/后端服务说明文档.md`）重新设计前端全部展示页面，包含展示内容与展示形式；
> 2. 沿用 aside + main 的双板块结构，但目录改为树状结构（含子目录），main 关联展示对应页面，内容全部重新设计；
> 3. 智能体输出全部耦合进具体业务页面，以仪表盘、状态环、链路图等可视化形态呈现，禁止独立的"智能体调度页"与任何对话框式输出。
>
> 本文档与 `frontend/DEV.md` 互为补充：DEV.md 记录当前真实状态与维护基线，本文档回答"接下来做成什么样、按什么顺序做"。落地后需同步更新 DEV.md。

## 1. 整体布局：aside + main 双板块（结构复用，内容重做）

```text
+----------------+--------------------------------------------------+
| aside 侧边栏    | main 主内容区                                     |
| (树状目录)      |  +--------------------------------------------+  |
|                |  | 顶部条：当前页面标题 / 全局时间范围选择器     |  |
| ▼ 总览驾驶舱    |  |          / 活跃预警角标(跳转预警中心)        |  |
| ▼ 日志监控      |  +--------------------------------------------+  |
|   ├ 应用服务    |  |                                              |  |
|   ├ 用户行为    |  |  <router-view> 渲染当前目录节点对应页面       |  |
|   ├ Web服务器   |  |                                              |  |
|   └ ...        |  +--------------------------------------------+  |
| ▼ 智能分析      |                                                  |
| ▼ 系统运维      |                                                  |
+----------------+--------------------------------------------------+
```

aside 设计要求：

1. 目录为**两级树**：一级为分组节点（可折叠，不可路由），二级为叶子节点（可路由）；总览驾驶舱是唯一的一级即叶子节点；
2. 目录数据由 `src/layout/menu.js` 以声明式数组维护（`{ title, icon, path?, children? }`），Layout 只负责渲染，**新增页面只改 menu.js 与路由，不改 Layout 模板逻辑**；
3. 当前路由命中的叶子节点高亮，其父分组自动展开；
4. aside 底部固定放置"系统链路健康"微型状态点（绿/黄/红），数据来自 `/system/status`，点击跳转系统运维页。

main 设计要求：

1. 顶部条全局复用：页面标题取自路由 meta；时间范围选择器（近 15 分钟 / 1 小时 / 6 小时 / 24 小时 / 自定义）通过 provide/inject 或简单的组合式函数 `useTimeRange()` 下发给页面，**所有图表与列表共用同一时间窗口**；
2. 活跃预警角标轮询 `GET /alerts/active`（30s），数字 > 0 时红点提示。

## 3. 各页面详细设计（内容与形式）

### 3.1 总览驾驶舱 `/dashboard`

定位：进入系统第一眼看到的"平台体检台"，回答"系统现在健不健康、业务量怎么样、有没有要处理的事"。

| 区块 | 展示内容 | 展示形式 |
|---|---|---|
| 健康总仪表 | 综合健康分（由错误率、链路状态、活跃预警数在前端按权重折算） | 大号仪表盘（gauge），绿/黄/红三段 |
| 核心指标带 | 日志总量、错误率、平均响应、P95 响应、活跃预警数 | 5 个数字卡，环比箭头（依赖双窗口对比接口，未就绪时隐藏箭头） |
| 流量趋势 | 按 interval 的请求量时间直方图（application + web_server） | 面积折线图，叠加错误量柱状 |
| 错误分布 | Top N 错误服务 / error_code 分布 | 横向条形图 + 环形图并排 |
| 耗时分布 | avg / p95 / p99 走势 | 多系列折线图 |
| 活跃预警摘要 | 最近 5 条 active 预警（类型、服务、时间） | 紧凑列表，点击跳转预警中心 |
| 最新体检结论 | 最近一份周期报告的 risk_level 与一句话摘要 | 小型风险仪表 + 摘要文字，点击跳转报告页（这是智能体产出在首页的**唯一**露出形式，见第 5 章） |

### 3.2 日志监控子页（7 页共用一套骨架，配置驱动差异）

7 个子页共用页面骨架 `LogMonitorShell`（页内组件，见第 4 章），骨架分三段：

```text
[筛选区]  时间窗口(全局) + log_type 固定 + 动态筛选字段(来自 /logs/fields) + 关键字
[图表带]  该日志类型专属的 2~3 个聚合图表
[明细区]  日志表格(分页) + 行展开查看完整字段 + 含 trace_id 的行可一键跳转链路追踪页
```

各子页差异通过一份页面级配置对象声明（图表模板、表格默认列、级别配色），不复制页面代码：

| 子页 | 图表带内容（对应后端聚合模板） | 明细表默认列 |
|---|---|---|
| 应用服务日志 | 错误率趋势（errors）、Top N 慢接口（latency）、error_code 分布 | 时间、级别、服务、接口、状态码、耗时、错误码 |
| 用户行为日志 | 行为类型分布（terms）、转化步骤计数（behavior_funnel 简化版）、热门商品/关键词 Top N | 时间、用户、action、页面、商品、停留时长 |
| Web服务器日志 | 状态码分布（errors）、request_time 趋势（latency）、Top N URI（traffic） | 时间、URI、状态码、耗时、upstream、UA |
| 性能指标日志 | 指标趋势按 metric_name 切换（infra_health）、p95 对比 | 时间、服务、指标名、指标值、主机 |
| 安全日志 | risk_level 分布、Top N 风险 IP、拦截次数趋势（security） | 时间、事件类型、风险级、IP、规则、状态 |
| 基础设施日志 | 组件健康比率、Kafka lag 趋势、资源使用趋势（infra_health） | 时间、组件、资源类型、状态、详情 |
| 审计日志 | 操作人 Top N、操作类型分布 | 时间、操作人、操作、对象、变更前后值 |

设计约束：

1. 筛选字段**必须**来自 `GET /logs/fields?log_type=xxx` 动态渲染，接口未就绪前用本地常量兜底并标注"待后端字段目录接口"；
2. 表格查询统一走 `POST /logs/search`（已有 `LogQueryRequest` 契约），分页、排序参数由骨架统一管理；
3. 每个子页的图表带在 ES 聚合接口未就绪时显示骨架屏 + "等待聚合接口"占位，明细表能力先行。

### 3.3 异常诊断中心 `/analysis/diagnosis`（智能体耦合页之一）

定位：把"规则引擎 + LangGraph 规则子图 + LLM 根因推断"的全部产出耦合为一个**事件作战面板**。页面上不出现"智能体""Agent""LLM"等调度字眼，用户视角只有"发起诊断 → 看到诊断结论"。

页面分四区：

```text
[输入区]   左侧窄栏：选择诊断入口（粘贴异常日志 / 选择活跃预警 / 填写服务+时间窗）
[结论区]   中部上方：严重度仪表 + 置信度仪表 + 异常类型徽章 + 根因结论卡
[证据区]   中部下方：证据时间轴 + 受影响服务关系图 + 同类错误频次迷你图
[建议区]   右侧窄栏：处理建议核对清单 + 分析阶段环
```

| 元素 | 内容 | 形式（必须可视化，禁止对话框） |
|---|---|---|
| 严重度仪表 | `severity`（low/medium/high/critical） | 四段式半环仪表，指针落位 + 颜色映射 |
| 置信度仪表 | `confidence`（0~1） | 环形进度环，中心显示百分数 |
| 异常类型徽章 | `anomaly_type` | 大号徽章 + 受影响服务标签组 |
| 根因结论卡 | `root_cause` 文本 | 固定结构卡片："结论一句话（加粗）+ 推理依据要点（列表）"，由前端对结构化字段做版式化渲染，**不渲染为聊天气泡** |
| 证据时间轴 | `evidence_logs` + 上下文日志 | 垂直时间轴，每条证据是一个节点（时间、服务、关键字段），ERROR 节点红色 |
| 服务关系图 | `affected_services` + trace 上下文 | 简化拓扑：服务节点连线，异常服务红色脉冲 |
| 同类错误频次 | `get_similar_errors` 结果 | 迷你柱状图（sparkline），展示"是否集中爆发" |
| 建议清单 | `suggestions` | 可勾选核对清单（勾选仅前端状态，用于演示处置过程） |
| 分析阶段环 | `node_trace`（取上下文→关联分析→根因推断→定级→报告） | **步骤环/阶段进度条**：每个阶段一个节点，显示耗时与状态（成功/降级/跳过），把图执行轨迹翻译为"分析进行到哪一步"，不出现节点函数名 |

降级展示：后端 LLM 不可用走纯规则结果时，结论卡显示"基于规则判定"角标，置信度仪表显示规则置信区间；阶段环将 LLM 阶段标记为"已跳过"。

### 3.4 周期体检报告 `/analysis/reports`（智能体耦合页之二）

定位：承接 LangGraph 定时子图每 15 分钟产出的周期报告。替代旧"实验结果"页。

```text
[左栏]  报告时间轴：按时间倒序的报告列表（时间、risk_level 色点、报告类型）
[右栏]  选中报告的结构化拆解
```

右栏拆解区块：

| 区块 | 内容 | 形式 |
|---|---|---|
| 风险定级 | 报告 `risk_level` | 三色风险仪表盘 + 定级理由摘要 |
| 窗口指标回放 | 报告所属窗口的流量/错误/耗时统计（来自报告内 metrics 快照） | 三个迷你趋势图横排，异常区间高亮 |
| 体检结论 | 报告正文（结构化字段） | 分节卡片：总体结论 / 异常发现 / 业务洞察，每节为"标题 + 要点列表"版式 |
| 隐藏关系发现 | relation_chain 产出（如"请求高峰后支付失败上升"） | 关系卡：左指标→右指标的箭头卡片，附两条指标的对照迷你折线图佐证 |
| 生成质量带 | `node_trace` 摘要 | 页脚一行阶段点（聚合→采样→压缩→分析→成文），降级生成的报告显示"统计模式"灰色角标 |

### 3.5 预警中心 `/analysis/alerts`

| 区块 | 内容 | 形式 |
|---|---|---|
| 状态看板 | active / acknowledged / resolved 三态计数 | 三个状态卡 + 24h 预警数量趋势迷你图 |
| 预警列表 | 类型、severity、受影响服务、首次/最近时间、evidence_count（去重累计） | 表格，severity 色块，active 行支持"确认"按钮（`POST /alerts/{id}/ack`） |
| 详情抽屉 | 选中预警的预警解释文案（alert_chain 产出）、关联证据、关联报告链接 | 右侧抽屉：解释文案按"现象/影响/建议"三段版式渲染；附"发起深度诊断"按钮跳转诊断中心并预填该预警 |

### 3.8 系统运维三页

| 页面 | 内容 | 形式 |

### 4.2 组件归属规则（强约束）

1. **一页一目录**：仅被单一页面使用的组件，必须放在 `components/<页面标识>/` 下，目录名与路由叶子一一对应（监控 7 子页因共用同一骨架，共享 `components/monitor/`，子页差异走配置对象而非组件复制）；
2. **通用组件准入**：组件进入 `components/common/` 的唯一条件是"已被 ≥2 个页面实际使用"；预判可能复用但当前只有一处使用的，先放页面目录，第二处出现时再上移；
3. **依赖方向单向**：`views → components/<page> → components/common → utils/api`，禁止页面目录组件互相引用、禁止 common 组件反向引用页面组件；
4. **图表只进不出**：所有 ECharts 实例只允许存在于 `components/common/charts/` 内，页面组件传 data/配置，不直接 import echarts；
5. **API 唯一通道**：保持 DEV.md 既有约束，页面与组件不得直接 axios/fetch。

## 5. 智能体展示的耦合规范（硬性规则）

本章是对需求第 3 点的统一约束，所有页面开发必须遵守：

1. **不设独立智能体页面**：目录树中不存在"智能体""Agent 调度""LLM 分析"等节点；LangGraph/LangChain 的产出只能通过诊断中心、体检报告、预警中心、总览驾驶舱四个业务页面露出；
2. **禁止对话框式输出**：智能体生成的任何文本（根因、报告正文、预警解释）不得以聊天气泡、消息流、打字机对话等形式渲染；必须先经过**版式化拆解**（结论句 / 要点列表 / 三段式"现象-影响-建议"等固定版式），并与至少一个可视化元素（仪表、趋势图、时间轴、关系卡）组合后呈现；
3. **数值先于文字**：severity、confidence、risk_level 等定级字段一律先以仪表/进度环/色块呈现，文字解释作为仪表的附注；
4. **调度过程翻译为业务进度**：`node_trace` 是唯一允许体现"图执行过程"的数据，且只能通过 `StageRing` 组件以"分析阶段"语义展示（如"取上下文 → 关联分析 → 根因推断 → 定级 → 成文"），不得展示节点函数名、工具名、模型名；
5. **降级可感知但不突兀**：LLM 降级产出的报告/诊断用灰色"统计模式 / 规则判定"角标标识，不弹窗、不报错；
6. **基础日志可视化对齐后端聚合能力**：所有图表的数据口径必须落在后端六类聚合模板与字段目录能力范围内，前端不自创聚合口径、不在浏览器端对原始日志做大规模二次统计（明细表行内的轻量格式化除外）。

## 8. 边界与未完成项声明

1. 本规划只涉及 `location/frontend/`，不修改 backend、docker-compose、elasticsearch、logstash、kibana、setup；
