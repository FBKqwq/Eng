# 后端服务说明文档

> 项目名称：基于 ELK、Kafka 与 LangGraph 的模拟电商实时日志分析与智能异常诊断系统  
> 文档对象：`backend/` 后端服务  
> 适用范围：用于说明当前项目后端涉及的服务模块、职责边界、调用关系与后续扩展方向。

---

## 1. 后端服务总体定位

当前项目采用“基础设施层 + 业务应用层”的双层结构：

```text
location/
├── elasticsearch/
├── kibana/
├── logstash/
├── setup/
├── docker-compose.yml
├── backend/
└── frontend/
```

其中，`elasticsearch/`、`kibana/`、`logstash/`、`setup/` 和 `docker-compose.yml` 主要承担基础设施运行职责；`backend/` 才是本项目真正承载业务逻辑、日志生产、数据查询和智能诊断能力的核心服务层。

后端不是单纯的接口转发服务，而是整个系统的业务中枢和智能分析中枢。它主要负责：

1. 对外提供 HTTP API；
2. 生成模拟电商业务日志；
3. 将日志写入 Kafka；
4. 查询 Elasticsearch 中的日志数据；
5. 执行规则分流与智能诊断；
6. 向前端返回结构化日志、状态和诊断结果。

## 3. 后端服务分层说明

后端整体采用分层设计，推荐按照以下调用方向组织代码：

```text
Frontend
   ↓
API Layer
   ↓
Service Layer
   ↓
Repository / External Client Layer
   ↓
Kafka / Elasticsearch / LLM / Other Infra
```

### 3.1 API 层

API 层位于：

```text
backend/app/api/
```

主要职责是：

1. 接收前端请求；
2. 进行基础参数校验；
3. 调用对应 service；
4. 返回统一响应结果。

API 层不应该写复杂业务逻辑，也不应该直接操作 Kafka 或 Elasticsearch。

推荐接口模块包括：

```text
api/v1/
├── health.py
├── logs.py
├── diagnosis.py
├── metrics.py
└── system.py
```

| 接口模块 | 服务职责 |
|---|---|
| `health.py` | 后端健康检查 |
| `logs.py` | 日志查询、筛选、分页 |
| `diagnosis.py` | 智能异常诊断 |
| `metrics.py` | 日志统计、指标聚合 |
| `system.py` | 系统运行状态检查 |

---

### 3.2 Schema 层

Schema 层位于：

```text
backend/app/schemas/
```

它是前后端协作的“接口合同”，用于定义请求体、响应体和字段校验规则。

推荐文件包括：

```text
schemas/
├── common.py
├── log.py
├── diagnosis.py
└── metrics.py
```

| 文件 | 说明 |
|---|---|
| `common.py` | 通用响应结构、分页结构、错误结构 |
| `log.py` | 日志查询请求、日志记录响应 |
| `diagnosis.py` | 智能诊断请求、诊断结果响应 |
| `metrics.py` | 统计指标请求与响应 |

Schema 层的价值在于：

1. 统一前后端字段；
2. 避免接口返回格式混乱；
3. 提高参数校验能力；
4. 便于后续生成接口文档；
5. 便于答辩时展示系统工程规范。

## 5. Kafka 日志生产服务

### 5.1 服务位置

```text
backend/app/services/kafka/
```

### 5.2 核心文件

```text
producer.py
topic_manager.py
```

### 5.3 服务职责

Kafka 服务主要负责将后端生成的日志消息写入 Kafka Topic。

核心职责包括：

1. 初始化 Kafka Producer；
2. 将日志数据序列化为 JSON；
3. 发送日志到指定 Topic；
4. 处理发送失败、重试和异常记录；
5. 可选支持 Topic 初始化和检查。

### 5.4 典型调用关系

```text
simulation/log_generator.py
        ↓
services/kafka/producer.py
        ↓
Kafka Topic
```

### 5.5 在项目中的作用

Kafka 日志生产服务是整个日志链路的入口之一。它负责把模拟电商业务产生的日志推送到消息队列中，使后续 Logstash 可以异步消费日志。

使用 Kafka 的核心价值是：

1. 解耦日志生产和日志处理；
2. 缓冲高并发日志写入；
3. 避免日志处理服务短暂异常时导致日志直接丢失；
4. 为后续扩展多个消费者提供基础。

## 6. Elasticsearch 查询服务

### 6.1 服务位置

```text
backend/app/services/elasticsearch/
```

### 6.2 核心文件

```text
client.py
log_query_service.py
index_service.py
```

### 6.3 服务职责

Elasticsearch 服务负责后端与 ES 的交互，包括日志检索、聚合分析和索引管理。

核心职责包括：

1. 创建 Elasticsearch 客户端；
2. 按条件查询日志；
3. 支持时间范围筛选；
4. 支持日志级别筛选；
5. 支持服务名、接口路径、错误码等字段筛选；
6. 支持日志聚合统计；
7. 支持索引初始化与 Mapping 管理。

### 6.4 典型查询场景

后端应支持以下查询能力：

| 查询类型 | 说明 |
|---|---|
| 最近日志查询 | 查询最近一段时间的日志 |
| 条件筛选查询 | 按日志级别、服务名、错误码等筛选 |
| Trace 查询 | 根据 `trace_id` 查询一次请求链路 |
| 错误日志查询 | 查询 `ERROR` 或异常状态码日志 |
| 聚合统计 | 统计错误率、接口耗时、服务分布等 |

### 6.5 在项目中的作用

Elasticsearch 查询服务是后端进行日志分析和智能诊断的证据来源。

对于智能诊断模块来说，不能只依赖用户输入的一条异常信息，而应该从 ES 中拉取相关上下文，例如：

1. 同一 `trace_id` 的完整调用链日志；
2. 同一服务最近 5 分钟内的异常日志；
3. 同一错误码的出现频率；
4. 同一接口的平均耗时和异常比例。

这些数据会作为智能诊断的证据输入。

## 7. 模拟业务与日志生成服务

### 7.1 服务位置

```text
backend/app/services/simulation/
```

### 7.2 核心文件

```text
log_generator.py
event_templates.py
mock_user_behavior.py
```

### 7.3 服务职责

模拟服务负责构造电商业务场景下的日志数据，是当前系统的数据源。

它应模拟的业务行为包括：

1. 用户浏览商品；
2. 用户搜索商品；
3. 用户点击推荐位；
4. 用户加入购物车；
5. 用户提交订单；
6. 用户支付；
7. 库存校验；
8. 登录认证；
9. 高频访问；
10. 接口异常。

### 7.4 日志类型

建议后端至少生成三类日志：

| 日志类型 | 说明 |
|---|---|
| 访问日志 | 用户访问接口、浏览页面、点击商品等行为 |
| 业务日志 | 下单、支付、库存校验等业务流程日志 |
| 异常日志 | 接口超时、支付失败、库存不足、服务错误等异常 |

### 7.5 推荐日志字段

```json
{
  "timestamp": "2026-05-06T19:00:00",
  "log_level": "ERROR",
  "service_name": "payment-service",
  "host": "server-01",
  "trace_id": "trace_20260506_001",
  "user_id": "user_10001",
  "session_id": "session_abc",
  "request_path": "/api/order/pay",
  "http_method": "POST",
  "status_code": 500,
  "response_time_ms": 3200,
  "event_type": "payment_failed",
  "error_code": "DB_TIMEOUT",
  "message": "payment service database timeout",
  "source_type": "backend",
  "env": "dev"
}
```

### 7.6 在项目中的作用

模拟日志生成服务决定了项目是否有足够真实的数据可供展示和分析。

如果日志设计过于简单，Kibana 图表和智能诊断都会显得空洞。因此，日志生成服务不应该只随机生成 `INFO`、`ERROR`，而应围绕电商业务链路构造有业务含义的日志。

## 8. 智能诊断服务

### 8.1 服务位置

```text
backend/app/services/diagnosis/
```

### 8.2 核心文件

```text
rule_engine.py
analyzer.py
langchain_service.py
langgraph_flow.py
```

### 8.3 服务职责

智能诊断服务是当前项目的创新核心，负责对异常日志进行规则判断、上下文检索、根因分析和建议生成。

整体流程如下：

```text
异常日志输入
   ↓
规则分流
   ↓
上下文检索
   ↓
LangGraph 诊断流程
   ↓
LangChain 调用大模型
   ↓
结构化诊断结果
```

---

### 8.4 rule_engine.py：规则分流服务

`rule_engine.py` 负责确定性异常的快速判断。

适合由规则直接识别的异常包括：

| 规则 | 诊断方向 |
|---|---|
| `status_code >= 500` | 服务端异常 |
| `response_time_ms > 3000` | 接口超时 |
| `error_code = DB_TIMEOUT` | 数据库超时 |
| `error_code = STOCK_NOT_ENOUGH` | 库存不足 |
| 短时间同 IP 高频访问 | 可疑访问 |
| 支付接口连续失败 | 支付链路异常 |

规则分流的价值在于：

1. 简单问题不交给大模型，降低成本；
2. 确定性异常直接返回，提高实时性；
3. 为复杂诊断提供初始标签；
4. 降低大模型幻觉风险。

---

### 8.5 analyzer.py：诊断总入口服务

`analyzer.py` 是智能诊断模块对外暴露的统一入口。

它负责协调：

1. 规则引擎；
2. Elasticsearch 日志上下文查询；
3. LangGraph 工作流；
4. 诊断结果结构化输出。

推荐职责如下：

```text
diagnosis.py API
      ↓
analyzer.py
      ↓
rule_engine.py
      ↓
log_query_service.py
      ↓
langgraph_flow.py
      ↓
langchain_service.py
```

`analyzer.py` 不应该写成一个巨大函数，而应该只负责流程协调。

---

### 8.6 langchain_service.py：大模型调用服务

`langchain_service.py` 负责封装大模型调用。

主要职责包括：

1. Prompt 模板管理；
2. 上下文日志拼接；
3. 模型 API 调用；
4. 输出格式约束；
5. JSON 结果解析；
6. 失败重试与异常兜底。

LangChain 在本项目中不负责整个流程编排，只负责“大模型调用与上下文组织”。

---

### 8.7 langgraph_flow.py：诊断流程编排服务

`langgraph_flow.py` 负责定义智能诊断工作流。

推荐最小流程如下：

```text
开始
 ↓
规则分流节点
 ↓
上下文检索节点
 ↓
异常分类节点
 ↓
根因分析节点
 ↓
建议生成节点
 ↓
结构化输出节点
```

每个节点只处理一个明确任务，这样有利于调试、展示和扩展。

LangGraph 的价值在于：

1. 明确诊断流程；
2. 管理节点状态；
3. 支持条件分支；
4. 便于后续扩展多种诊断路径；
5. 比单次 LLM 调用更具工程可解释性。

---

### 8.8 诊断结果结构

智能诊断接口建议返回如下结构：

```json
{
  "anomaly_type": "支付链路超时",
  "severity": "high",
  "root_cause": "payment-service 下游数据库查询延迟升高，导致订单提交链路超时",
  "affected_services": ["order-service", "payment-service"],
  "time_range": "2026-05-06 19:00:00 ~ 2026-05-06 19:05:00",
  "evidence_logs": [
    "payment-service response_time_ms > 3000",
    "error_code=DB_TIMEOUT 出现 28 次",
    "trace_id 在 order-service 到 payment-service 链路中断"
  ],
  "suggestions": [
    "检查 payment-service 数据库连接池",
    "排查数据库慢查询",
    "必要时临时降级支付接口"
  ],
  "confidence": 0.87
}
```

## 9. 系统状态服务

### 9.1 服务位置

```text
backend/app/api/v1/system.py
```

也可在服务层补充：

```text
backend/app/services/system/
```

### 9.2 服务职责

系统状态服务用于检查当前后端依赖的基础设施是否可用。

建议检查内容包括：

| 检查项 | 说明 |
|---|---|
| Backend | FastAPI 是否正常运行 |
| Kafka | Kafka 地址、Topic 是否可用 |
| Elasticsearch | ES 是否可访问，集群状态是否正常 |
| Kibana | Kibana 地址展示 |
| Logstash | 管道状态可选检查 |
| LLM API | 模型服务是否配置 |

### 9.3 在项目中的作用

系统状态服务主要服务于前端“系统状态”页面，也适合答辩时展示系统连通性。

它可以帮助说明：

1. 当前各基础设施是否正常；
2. 后端是否能连接 Kafka 和 ES；
3. 项目是否真正完成了前后端与 ELK 的整合。

## 10. 指标统计服务

### 10.1 服务位置

```text
backend/app/services/dashboard/
```

或：

```text
backend/app/services/metrics/
```

### 10.2 服务职责

指标统计服务用于为前端首页、监控页和实验结果页提供汇总数据。

推荐统计指标包括：

| 指标 | 说明 |
|---|---|
| 日志总量 | 当前索引中的日志数量 |
| 错误日志数量 | `ERROR` 级别日志数量 |
| 错误率 | 错误日志 / 日志总量 |
| 平均响应时间 | 接口平均耗时 |
| P95 响应时间 | 95 分位接口耗时 |
| Top N 异常服务 | 错误最多的服务 |
| Top N 慢接口 | 响应时间最高的接口 |
| 日志时间趋势 | 按时间聚合日志数量 |

### 10.3 在项目中的作用

指标统计服务用于把 Elasticsearch 中的原始日志转化为更适合展示的业务指标。

Kibana 适合做专业日志可视化，而前端页面更适合展示简化后的摘要结果。因此，后端需要提供指标聚合接口，避免前端直接访问 Elasticsearch。

## 13. 后端 API 服务建议

后端建议提供以下接口：

| 方法 | 路径 | 功能 |
|---|---|---|
| `GET` | `/api/v1/health` | 健康检查 |
| `POST` | `/api/v1/logs/search` | 日志检索 |
| `GET` | `/api/v1/logs/recent` | 最近日志 |
| `GET` | `/api/v1/logs/trace/{trace_id}` | 调用链日志查询 |
| `GET` | `/api/v1/metrics/summary` | 指标概览 |

## 14. 后端服务与基础设施的关系

后端并不替代 Kafka、Logstash、Elasticsearch 和 Kibana，而是对它们进行业务化封装。

| 基础设施 | 后端如何使用 |
|---|---|
| Kafka | 后端作为 Producer 写入日志 |
| Logstash | 后端不直接调用，由 Logstash 消费 Kafka |
| Elasticsearch | 后端作为查询客户端读取日志和聚合结果 |
| Kibana | 后端不直接依赖，Kibana 独立做可视化 |
| LangChain | 后端封装模型调用 |
| LangGraph | 后端编排诊断流程 |

后端的作用可以理解为：

```text
把基础设施能力包装成业务系统可以使用的 API。
```

## 16. 总结

当前项目后端的核心价值不只是提供接口，而是把 Kafka、Elasticsearch、规则引擎和智能诊断流程组织成一个统一的业务服务层。

从工程结构上看，后端主要承担四类能力：

1. **日志生产能力**：模拟电商行为并写入 Kafka；
2. **日志查询能力**：从 Elasticsearch 检索和聚合日志；
3. **智能诊断能力**：基于规则分流、上下文日志和 LangGraph/LangChain 输出诊断结果；
4. **系统支撑能力**：提供健康检查、系统状态、任务脚本和配置管理。

因此，`backend/` 是整个项目从“ELK 实验链路”升级为“日志分析与智能诊断系统”的关键部分。
